**📘 РАЗДЕЛ 3: ЕДИНАЯ МОДЕЛЬ КОНТЕНТА И СИСТЕМА РАБОТЫ С БЛОКАМИ**

**3.1 Философия единого контента: «Всё — блок, всё — настраиваемо»**

GlubLink отвергает традиционное разделение на «файлы», «заметки» и «ссылки». Вместо этого используется единая абстракция — **Контент-блок**. Тип блока определяет лишь набор дополнительных полей и способ визуального представления, но не влияет на базовые операции: поиск, тегирование, привязку к Workspace, экспорт.

**Мандаты модели:**

1.  **Единый интерфейс операций.** Любое действие (добавить тег, закрепить, связать, экспортировать) применяется одинаково ко всем типам блоков.
2.  **Независимость от физического расположения.** Блок существует в приложении независимо от того, хранится ли его медиа-данные в файловой системе, в заметке или во внешнем ресурсе.
3.  **Реактивная связность.** Изменение метаданных одного блока мгновенно отражается во всех связанных представлениях (галерея, заметка, Workspace).
4.  **Обратимость изменений.** Любое изменение блока (кроме физического удаления файла по явному подтверждению) может быть отменено в пределах сессии или через историю изменений.

**3.2 Базовая структура Контент-блока**

Каждый блок в системе содержит обязательный набор полей, определяющих его идентичность, состояние и связи. Эти поля присутствуют у всех типов блоков без исключения.

**Обязательные поля (ядро блока):**

*   **Идентификатор:** Уникальный UUID (версия 4), генерируемый при создании. Не изменяется в течение жизненного цикла.
*   **Тип сущности:** Одно из значений: Media, Note, Link, Collection. Определяет доступные дополнительные поля и способ рендеринга.
*   **Дата создания:** Момент первого появления блока в системе (не дата файла на диске).
*   **Дата добавления:** Момент, когда блок был импортирован или создан пользователем.
*   **Дата последнего изменения:** Обновляется при любом изменении метаданных блока.
*   **Список тегов:** Массив идентификаторов тегов, привязанных к блоку. Порядок не имеет значения.
*   **Список связей:** Массив объектов, описывающих связи с другими блоками (целевой блок, тип связи, метаданные связи).
*   **Флаги состояния:**
    *   isFavorite: Блок помечен как избранный.
    *   isArchived: Блок перемещён в архив (скрыт из обычных представлений).
    *   isInRecycleBin: Блок находится в корзине (подлежит удалению через 24 часа).
    *   isHidden: Блок скрыт вручную пользователем.
*   **Мета-данные безопасности:**
    *   contentHash: SHA-256 хеш содержимого (для медиа) или текста (для заметок). Используется для обнаружения дубликатов.
    *   sourcePath: Исходный путь к файлу (для медиа) или URL (для ссылок). Может быть пустым для чисто текстовых заметок.
    *   integrityFlag: Флаг целостности данных (валиден/повреждён/отсутствует).

**Правила валидации ядра:**

1.  Идентификатор должен быть уникальным в пределах всей базы данных.
2.  Дата создания не может быть позже даты добавления.
3.  Список тегов не может содержать дубликаты.
4.  Связи должны ссылаться на существующие блоки (проверка при сохранении).
5.  Флаги состояния взаимно исключают друг друга: блок не может одновременно находиться в архиве и в корзине.

**3.3 Специфика типов блоков**

Каждый тип блока расширяет базовую структуру тип-специфичными полями. Эти поля определяют способ хранения, отображения и обработки контента.

**3.3.1 Блок типа «Медиа»**

Представляет изображение, анимацию или видеофайл.

**Дополнительные поля:**

*   **Физические характеристики:**
    *   width, height: Разрешение в пикселях.
    *   duration: Длительность в миллисекундах (для видео и анимаций, null для статичных изображений).
    *   fileSize: Размер файла в байтах.
    *   format: Расширение файла (jpg, png, gif, mp4, webm, и т.д.).
*   **Визуальные данные:**
    *   thumbnailPath: Путь к кэшированному превью (оптимизированное изображение для отображения в сетке).
    *   orientation: Ориентация (landscape, portrait, square).
*   **Метаданные файла:**
    *   exifData: Словарь метаданных EXIF (для изображений): дата съёмки, камера, настройки.
    *   colorPalette: Доминирующие цвета (массив из 5 значений в формате hex).
*   **Состояние обработки:**
    *   aiProcessingStatus: Статус обработки ИИ (pending, processing, completed, failed).
    *   embeddingVector: Векторное представление изображения (512 значений) для визуального поиска.

**Правила обработки медиа-блоков:**

1.  При импорте файла система автоматически извлекает физические характеристики и генерирует превью.
2.  Превью хранится в отдельном кэше с лимитом размера; при переполнении удаляются наименее используемые.
3.  Вектор эмбеддинга генерируется асинхронно; до его готовности блок доступен для ручного тегирования.
4.  Изменение исходного файла вне приложения помечает блок как integrityFlag: missing; пользователь получает уведомление.

**3.3.2 Блок типа «Заметка»**

Представляет текстовый контент с возможностью вложения медиа-блоков.

**Дополнительные поля:**

*   **Текстовое содержимое:**
    *   content: Текст в формате, совместимом с Markdown (поддерживает заголовки, списки, ссылки, код).
    *   wordCount: Количество слов (вычисляется автоматически при сохранении).
    *   lastEditedBy: Идентификатор устройства, внёсшего последнее изменение (для синхронизации).
*   **Вложенные блоки:**
    *   embeddedBlocks: Массив идентификаторов медиа-блоков, встроенных в заметку. Порядок отображения сохраняется.
*   **Структурные метаданные:**
    *   headings: Автоматически извлечённый список заголовков (для навигации внутри заметки).
    *   links: Массив внешних ссылок, найденных в тексте (для предпросмотра).

**Правила обработки заметок:**

1.  Автосохранение происходит каждые 30 секунд при редактировании или при потере фокуса полем ввода.
2.  Вложенные медиа-блоки отображаются как превью с возможностью перехода к полному просмотру.
3.  При экспорте заметки вложенные медиа могут быть включены как ссылки или как встроенные данные (в зависимости от формата экспорта).
4.  Поиск по заметкам учитывает как текст содержимого, так и теги вложенных медиа-блоков.

**3.3.3 Блок типа «Ссылка»**

Представляет внешний ресурс с метаданными для быстрого доступа.

**Дополнительные поля:**

*   **Данные ресурса:**
    *   url: Полный URL ресурса (валидируется при сохранении).
    *   title: Заголовок страницы (извлекается из Open Graph или <title>).
    *   description: Краткое описание (из Open Graph или первые 200 символов контента).
    *   faviconPath: Путь к кэшированному значку сайта.
    *   previewImagePath: Путь к кэшированному превью страницы (Open Graph image).
*   **Состояние доступа:**
    *   lastChecked: Дата последней проверки доступности ресурса.
    *   httpStatus: Последний полученный HTTP-статус (200, 404, 503, и т.д.).
    *   isAccessible: Флаг доступности ресурса на момент последней проверки.

**Правила обработки ссылок:**

1.  При добавлении ссылки система асинхронно извлекает метаданные; до завершения блок отображается с заглушкой.
2.  Проверка доступности ресурса выполняется при каждом открытии блока и фоновом обновлении раз в 24 часа.
3.  При изменении заголовка или описания на исходной странице пользователь получает уведомление о доступном обновлении.
4.  Кэш превью и фавиконов имеет TTL 30 дней; устаревшие данные обновляются при следующем доступе.

**3.3.4 Блок типа «Коллекция»**

Виртуальная группа блоков, определяемая правилами фильтрации или ручным выбором.

**Дополнительные поля:**

*   **Правила выборки:**
    *   filterRules: Объект, описывающий условия включения блоков (теги, типы, даты, метаданные).
    *   excludeRules: Объект, описывающий условия исключения блоков.
    *   logicalOperator: Оператор объединения правил (AND, OR).
*   **Параметры отображения:**
    *   viewMode: Предпочтительный режим отображения (masonry, justified, list, timeline).
    *   sortOrder: Правило сортировки (dateAdded, dateCreated, tagName, popularity).
    *   grouping: Правило группировки (none, byTag, byDate, byType).
*   **Ручные закрепления:**
    *   pinnedBlocks: Массив идентификаторов блоков, закреплённых в коллекции вручную (имеют приоритет над правилами).
    *   pinnedOrder: Порядок отображения закреплённых блоков.

**Правила обработки коллекций:**

1.  Коллекция не хранит блоки физически; она вычисляет принадлежность на лету или кэширует результат выборки.
2.  Изменение правил коллекции мгновенно обновляет её содержимое во всех открытых представлениях.
3.  Ручное закрепление блока в коллекции не влияет на его принадлежность к другим коллекциям или тегам.
4.  При экспорте коллекции экспортируются только правила и закреплённые блоки; результат выборки может быть сгенерирован заново.

**3.4 Жизненный цикл блока**

Каждый блок проходит через определённые состояния от создания до архивации или удаления. Система гарантирует атомарность переходов и обратимость изменений.

**Этапы жизненного цикла:**

1.  **Создание:**
    *   Генерируется UUID, устанавливаются даты, инициализируются пустые списки тегов и связей.
    *   Для медиа-блоков: асинхронно извлекаются метаданные и генерируется превью.
    *   Блок становится доступен для поиска и отображения немедленно после базовой инициализации.
2.  **Активное использование:**
    *   Пользователь может изменять теги, связи, флаги состояния, тип-специфичные поля.
    *   Каждое изменение создаёт запись в журнале изменений (Change Log) для возможности отката.
    *   Изменения реактивно распространяются на все представления, где используется блок.
3.  **Архивация:**
    *   Блок помечается флагом isArchived и скрывается из обычных представлений.
    *   Архивированные блоки доступны через специальный фильтр или прямой поиск по идентификатору.
    *   Архивация обратима: снятие флага возвращает блок в активное состояние.
4.  **Корзина:**
    *   Блок помечается флагом isInRecycleBin и перемещается в специальную системную коллекцию.
    *   Блоки в корзине автоматически удаляются через 24 часа после помещения (настраивается в настройках).
    *   Пользователь может вручную восстановить блок из корзины или удалить его окончательно.
5.  **Физическое удаление:**
    *   Применяется только к медиа-блокам по явному подтверждению пользователя.
    *   Удаляется файл с диска, запись из базы данных, кэшированные превью и эмбеддинги.
    *   Все связи с другими блоками разрываются; связанные блоки получают уведомление о разрыве.

**Правила переходов:**

*   Переход в корзину возможен только из активного состояния или архива.
*   Восстановление из корзины возвращает блок в состояние, предшествовавшее удалению.
*   Физическое удаление необратимо; система запрашивает подтверждение и предлагает создать резервную копию.
*   Изменение типа блока (например, из «Заметки» в «Медиа») не поддерживается; необходимо создать новый блок и перенести данные.

**3.5 Система связей между блоками**

Блоки могут быть связаны между собой для создания семантических отношений. Связи хранятся отдельно от данных блоков и позволяют строить графы контента.

**Структура связи:**

*   **Идентификатор связи:** Уникальный UUID.
*   **Источник и цель:** Идентификаторы двух блоков, участвующих в связи.
*   **Тип связи:** Одно из предопределённых значений:
    *   reference: Блок А является референсом для блока Б.
    *   variation: Блоки представляют вариации одного объекта (персонаж, сцена).
    *   partOf: Блок А является частью блока Б (например, кадр из видео).
    *   inspiredBy: Блок А вдохновлён блоком Б.
    *   contradicts: Блоки содержат противоречащую информацию.
    *   custom: Пользовательский тип связи с произвольным названием.
*   **Метаданные связи:**
    *   description: Текстовое описание связи (опционально).
    *   strength: Числовая оценка силы связи (0.0–1.0, используется для ранжирования).
    *   createdAt: Дата создания связи.
    *   createdBy: Идентификатор устройства, создавшего связь.

**Правила управления связями:**

1.  Связи являются двунаправленными: создание связи А→Б автоматически создаёт обратную связь Б→А с тем же типом.
2.  Удаление связи с одной стороны удаляет её полностью; система уведомляет пользователя о разрыве.
3.  Связи не влияют на принадлежность блоков к коллекциям или тегам; это независимые механизмы организации.
4.  При экспорте блока связи могут быть включены как метаданные (в форматах, поддерживающих графы).

**Визуализация связей:**

*   В интерфейсе связи отображаются как интерактивные линии между блоками в специальном режиме «Граф».
*   При наведении на связь показывается её тип и описание.
*   Пользователь может фильтровать граф по типам связей или силе связи.
*   Режим графа поддерживает масштабирование, панорамирование и выделение подграфов.

**3.6 Взаимодействие блоков с системой тегов**

Теги являются основным механизмом категоризации и поиска блоков. Система обеспечивает гибкое управление тегами без дублирования данных.

**Правила привязки тегов:**

1.  Блок может иметь любое количество тегов; теги не иерархичны по умолчанию (плоская структура).
2.  Привязка тега к блоку не создаёт копию тега; хранится только ссылка на идентификатор тега.
3.  Изменение метаданных тега (название, категория, описание) мгновенно отражается во всех привязанных блоках.
4.  Удаление тега из системы разрывает все привязки; пользователь получает уведомление и возможность заменить тег на другой.

**Операции с тегами на уровне блока:**

*   **Добавление тега:** Мгновенно обновляет индекс поиска и представления, где используется блок.
*   **Удаление тега:** Обратимо в пределах сессии; запись в журнале изменений позволяет восстановить привязку.
*   **Замена тега:** Атомарная операция удаления старого и добавления нового тега; сохраняет историю изменений.
*   **Массовое тегирование:** Применение тега к нескольким блокам выполняется в транзакции; при ошибке откатывается полностью.

**Визуальные индикаторы тегов:**

*   В интерфейсе теги отображаются как цветные чипы; цвет определяется категорией тега.
*   При наведении на чип показывается описание тега и количество блоков, к которым он привязан.
*   Клик по чипу в контексте блока открывает поиск по этому тегу; клик в контексте поиска добавляет/удаляет тег из фильтра.
*   Защищённые мета-теги (image, gif, video) отображаются с иконкой замка и не могут быть удалены пользователем.

**3.7 Жёсткие правила генерации кода (Модель данных)**

Данные требования являются обязательными для всех последующих этапов реализации. Любое отклонение считается архитектурной ошибкой.

1.  **Единый базовый класс для всех блоков.** Все типы блоков наследуются от абстрактного класса ContentBlock, который определяет обязательные поля и интерфейсы операций. Запрещено дублировать код ядра блока в тип-специфичных реализациях.
2.  **Строгая типизация полей.** Все поля модели должны иметь явные типы; запрещено использование динамических типов (dynamic) в доменных сущностях. Перечисления (типы блоков, типы связей, флаги) описываются через enum.
3.  **Атомарность операций с блоками.** Любое действие, изменяющее состояние блока (добавление тега, создание связи, изменение флага), должно выполняться в транзакции. При ошибке система откатывает все изменения и логирует причину.
4.  **Реактивное обновление представлений.** Изменение данных блока должно автоматически уведомлять все подписанные виджеты. Запрещено ручное обновление UI через setState для данных; только через реактивные провайдеры.
5.  **Изоляция побочных эффектов.** Операции ввода-вывода (чтение файлов, запросы к сети, генерация эмбеддингов) должны быть инкапсулированы в репозиториях. Доменные сущности не содержат кода, зависящего от фреймворков или платформ.
6.  **Валидация на границе домена.** Все входящие данные (при импорте, создании, редактировании) должны проходить валидацию до сохранения в базу. Ошибки валидации возвращаются как описательные объекты, а не исключения.
7.  **Поддержка миграций схемы.** Изменения в структуре модели должны сопровождаться миграционными скриптами. Система должна корректно обрабатывать загрузку данных, созданных в предыдущих версиях схемы.
8.  **Тестируемость модели.** Каждая сущность и операция должны быть изолированы от внешних зависимостей. Это позволяет запускать юнит-тесты без инициализации базы данных или файловой системы.
