*   Лицензиар предоставляет Лицензиату неисключительное, непередаваемое право использовать Программу исключительно для личных, некоммерческих целей

**Интеллектуальная собственность:**

*   Все права на код, дизайн, архитектуру и документацию принадлежат Автору

*   Лицензиат использует Программу на свой страх и риск

**Прекращение и применимое право:**

*   Лицензия автоматически прекращается при нарушении условий использования
*   При прекращении Лицензиат обязан удалить все копии Программы со своих устройств

**14.3.2 Принципы обработки данных**

**Принцип минимизации:**

*   Собирается только то, что необходимо для работы приложения
*   Пути к файлам хранятся как строки, без сканирования содержимого

**Принцип локальности:**

*   Все пользовательские данные хранятся только на устройстве пользователя
*   Облачная синхронизация является опциональной и использует сквозное шифрование

**Принцип прозрачности:**

*   Пользователь видит, какие данные и зачем собираются
*   Настройки приватности доступны в один клик из главного меню

**Принцип контроля:**

*   Пользователь может экспортировать или удалить все данные в любой момент
*   Аналитика и отчёты об ошибках отключаются по желанию пользователя

**14.3.3 Политика конфиденциальности**

Политика конфиденциальности должна быть доступна пользователю в понятной форме и содержать:

**Что приложение не делает:**

*   Не продаёт и не передаёт пользовательские данные третьим лицам
*   Не сканирует содержимое файлов без явного запроса пользователя
*   Не отслеживает активность пользователя вне приложения

**Что приложение делает:**

*   Хранит метаданные медиа локально для обеспечения быстрого поиска
*   Использует внешние сервисы только для функций, которые пользователь явно включает
*   Отправляет анонимные отчёты об ошибках для улучшения стабильности (с возможностью отключения)

**Права пользователя:**

*   Экспортировать все свои данные в любой момент
*   Полностью удалить все данные приложения одной кнопкой
*   Отключить любую интеграцию с внешними сервисами

**14.5.2 Реализация соответствия требованиям**

**Обязательные заголовки запросов:**

**Проверка лимитов перед запросом:**

*   Перед каждым запросом к внешнему сервису должна выполняться проверка текущего количества запросов
*   При приближении к лимиту запросы должны ставиться в очередь с экспоненциальной задержкой
*   Пользователь должен получать понятное уведомление при достижении лимита

**Логирование запросов для аудита:**

*   Логи не должны содержать персональные данные, токены доступа или пути к файлам пользователя
*   Логи хранятся локально с ограниченным сроком жизни для целей отладки

**14.5.3 Управление пользовательскими ключами внешних сервисов**

**Настройки интеграций:**

*   Ключи хранятся в защищённом хранилище операционной системы, не передаются автору приложения
*   Для каждого ключа должна быть возможность тестирования соединения и удаления

**Предупреждения для пользователя:**

*   Использование сторонних сервисов регулируется их условиями
*   Приложение не несёт ответственности за блокировку ключей или лимиты, установленные провайдерами
*   Пользователь должен явно согласиться на использование внешнего сервиса перед первым запросом

**14.6 Хранение и защита секретов**

**14.6.1 Архитектура безопасного хранения**

**Уровень один: Код приложения**

*   Никаких жёстко закодированных ключей или токенов в исходном коде
*   Только публичные конечные точки и конфигурации

**Уровень два: Защищённое хранилище операционной системы**

*   Ключи внешних сервисов пользователя хранятся в системном хранилище учётных данных
*   Шифрование через механизмы операционной системы
*   Доступ только после аутентификации пользователя в системе

**Уровень три: Локальная база данных**

*   Настройки, теги, метаданные хранятся в локальной базе
*   Шифрование базы данных опционально, по выбору пользователя

**14.6.2 Реализация безопасного хранения**

**Запись ключа:**

*   Ключи записываются в защищённое хранилище с привязкой к имени сервиса
*   Операция должна возвращать статус успеха или ошибки для информирования пользователя

**Чтение ключа:**

*   Чтение ключа происходит только при необходимости выполнения запроса к сервису
*   Ключ не кэшируется в памяти дольше времени выполнения операции

**Удаление ключа:**

*   Пользователь может удалить сохранённый ключ в любой момент
*   После удаления все функции, зависящие от этого ключа, переходят в режим ограниченной функциональности

**Проверка наличия:**

*   Перед использованием ключа выполняется проверка его наличия в хранилище
*   При отсутствии ключа пользователю предлагается ввести его или использовать ограниченный режим

**14.6.3 Маскировка секретов в логах**

**Паттерны для маскировки:**

*   Строки, соответствующие формату ключей доступа, должны автоматически маскироваться перед записью в лог
*   Токены авторизации, пароли и другие чувствительные данные не должны попадать в логи в открытом виде

**Функция санитизации:**

*   Все сообщения, передаваемые в систему логирования, должны проходить через функцию очистки
*   Функция заменяет чувствительные данные на символы-заполнители перед записью

**Уровни логирования:**

*   В режиме разработки допускается более подробное логирование для отладки
*   В релизной сборке чувствительные данные маскируются всегда, независимо от уровня логирования

**14.7 Аудит и соответствие**

**14.7.1 Чек-лист перед релизом**

**Юридическая проверка:**

*   Файл лицензии присутствует и соответствует текущей версии приложения
*   Политика конфиденциальности обновлена с учётом новых функций
*   Список зависимостей сгенерирован актуальным списком пакетов
*   Все интеграции с внешними сервисами имеют пользовательское согласие в интерфейсе
*   В коде отсутствуют жёстко закодированные ключи или токены

**Техническая проверка:**

*   Анализ зависимостей не показывает пакетов с несовместимыми лицензиями
*   Все запросы к внешним сервисам имеют корректные заголовки пользователя
*   Логи в релизной сборке не содержат путей к файлам, токенов, персональных данных
*   Шифрование хранилища учётных данных работает на целевой операционной системе

**Пользовательская проверка:**

*   Настройки приватности доступны и понятны для пользователя
*   Функции экспорта и удаления данных работают корректно
*   Атрибуции сторонним сервисам отображаются в соответствующих разделах интерфейса

**14.7.2 Автоматизированные проверки в конвейере сборки**

**Проверка наличия файла лицензии:**

*   Конвейер сборки должен проверять наличие файла лицензии в корне проекта
*   Отсутствие файла блокирует создание релизной сборки

**Проверка на наличие жёстко закодированных секретов:**

*   Скрипт анализа кода должен искать паттерны, соответствующие ключам доступа
*   Обнаружение таких паттернов блокирует слияние кода в основную ветку

**Валидация лицензий зависимостей:**

*   Перед сборкой выполняется анализ всех зависимостей на предмет совместимости лицензий
*   Зависимости с несовместимыми лицензиями блокируют процесс сборки

**Генерация документации зависимостей:**

*   Автоматическая генерация актуального списка зависимостей с лицензиями
*   Документация включается в артефакты сборки для проверки перед релизом

**14.8 Чек-лист реализации**

**Этап один: Базовая инфраструктура**

1.  Создать файлы лицензии, политики конфиденциальности и списка зависимостей в корне проекта
2.  Настроить интеграцию с защищённым хранилищем операционной системы для ключей
3.  Реализовать функцию маскировки чувствительных данных в логах
4.  Добавить проверку лицензий зависимостей в конвейер непрерывной интеграции
5.  Вынести заголовки пользователя для внешних сервисов в константы конфигурации

**Этап два: Пользовательский интерфейс**

1.  Добавить экран «О приложении» с отображением лицензии и атрибуций сторонним компонентам
2.  Реализовать настройки «Приватность» с переключателями аналитики и отчётов об ошибках
3.  Добавить кнопку «Экспортировать мои данные» для создания архива с пользовательской информацией
4.  Добавить кнопку «Удалить все данные» с подтверждением и созданием резервной копии
5.  Реализовать ввод пользовательских ключей внешних сервисов с валидацией и тестированием

**Этап три: Аудит и документация**

1.  Провести ручной аудит кода на наличие жёстко закодированных секретов
2.  Протестировать шифрование хранилища учётных данных на целевой операционной системе
3.  Проверить, что логи в релизной сборке не содержат персональных данных
4.  Обновить политику конфиденциальности с учётом всех точек сбора данных
5.  Подготовить шаблон ответа на запросы по законодательству о персональных данных

**Этап четыре: Релиз и мониторинг**

1.  Включить проверку соответствия в обязательные шаги конвейера сборки
2.  Настроить уведомления на ошибки, связанные с доступом к защищённому хранилищу
3.  Добавить метрику «Пользователи с включённой аналитикой» в анонимную статистику
4.  Подготовить публичную страницу с документацией по лицензированию
5.  Собрать релизную сборку с включёнными проверками соответствия

**15.2 Уровни тестирования и их назначение**

**15.2.1 Пирамида тестов**

Тестирование строится по принципу пирамиды: большое количество быстрых изолированных тестов на нижнем уровне, меньше интеграционных тестов в середине, и минимальное количество полных сценариев на вершине.

**Уровень один: Модульные тесты**

*   **Назначение:** Проверка изолированной бизнес-логики без внешних зависимостей
*   **Объекты тестирования:** Функции парсинга поисковых запросов, алгоритмы сортировки тегов, валидация входных данных, расчёт путей организации файлов, логика векторных часов
*   **Требования к покрытию:** Не менее восьмидесяти процентов строк бизнес-логики, сто процентов критических путей
*   **Изоляция зависимостей:** Все внешние вызовы заменяются моками или стабами
*   **Критерии успеха:** Тест выполняется менее чем за одну секунду, не зависит от порядка выполнения других тестов, даёт детерминированный результат

**Уровень два: Компонентные тесты**

*   **Назначение:** Проверка отрисовки и поведения отдельных интерфейсных компонентов
*   **Объекты тестирования:** Карточки медиа, чипы тегов, панели навигации, поля ввода с автодополнением, диалоговые окна
*   **Проверяемые аспекты:** Корректная отрисовка в разных темах, реакция на пользовательские события, доступность для вспомогательных технологий, анимации переходов между состояниями
*   **Визуальные регрессионные тесты:** Для ключевых компонентов создаются эталонные изображения; любое изменение рендеринга требует явного подтверждения
*   **Критерии успеха:** Компонент корректно отображается при разных размерах окна, поддерживает навигацию с клавиатуры, озвучивается вспомогательными технологиями

**Уровень три: Интеграционные тесты**

*   **Назначение:** Проверка взаимодействия между слоями приложения и корректности сквозных сценариев
*   **Объекты тестирования:** Импорт папки, сканирование, запись в базу, отображение в галерее; применение тега, обновление индекса, фильтрация результатов; синхронизация изменения между двумя устройствами
*   **Тестовое окружение:** Используется изолированная тестовая база данных, временные директории для файлов, моки для внешних сервисов
*   **Проверяемые аспекты:** Атомарность транзакций, корректность обработки ошибок, сохранение состояния при прерывании, согласованность данных между слоями
*   **Критерии успеха:** Сценарий выполняется от начала до конца без ручного вмешательства, все промежуточные состояния соответствуют ожиданиям, данные сохраняются корректно

**Уровень четыре: Сквозные тесты**

*   **Назначение:** Проверка полных пользовательских сценариев в среде, максимально приближенной к продакшену
*   **Объекты тестирования:** Онбординг нового пользователя, поиск и фильтрация коллекции, полноэкранный просмотр с навигацией, создание и редактирование заметок, настройка синхронизации
*   **Тестовые данные:** Используются реалистичные наборы данных от одной тысячи до ста тысяч файлов, включая различные типы медиа, теги и связи
*   **Проверяемые аспекты:** Время отклика интерфейса, плавность анимаций, потребление памяти, корректность восстановления после сбоев
*   **Критерии успеха:** Пользователь достигает цели сценария без ошибок, производительность соответствует целевым метрикам, приложение не завершается аварийно

**15.3 Тестирование устойчивости: Инженерия хаоса**

Приложение должно оставаться стабильным не только в идеальных условиях, но и при возникновении непредвиденных ситуаций. Тестирование устойчивости намеренно создаёт сбои для проверки механизмов восстановления.

**15.3.2 Принципы тестирования устойчивости**

1.  **Автоматизация сценариев.** Каждый сценарий сбоя оформляется как воспроизводимый тест с чёткими критериями успеха.
2.  **Изоляция воздействия.** Сбои симулируются только в тестовом окружении; продакшен-сборки не содержат кода для искусственного создания ошибок.
3.  **Логирование для отладки.** Все реакции на сбои подробно логируются с контекстом, достаточным для воспроизведения и анализа.
4.  **Постепенное усложнение.** Тесты начинаются с простых сценариев и постепенно добавляют комбинации сбоев для проверки устойчивости к каскадным ошибкам.

3.  **Анонимная телеметрия в продакшене (с согласия).** Пользователи могут разрешить сбор агрегированных метрик производительности для выявления проблем на реальных конфигурациях.
4.  **Регулярное нагрузочное тестирование.** Еженедельный прогон сценариев с увеличенной нагрузкой для выявления деградации со временем.

**15.6 Управление тестовыми данными и окружением**

Эффективное тестирование требует изолированного окружения и реалистичных, но безопасных тестовых данных.

**15.6.1 Стратегия управления тестовыми данными**

*   **Фикстуры для быстрых тестов.** Небольшие наборы данных для модульных и виджет-тестов; хранятся в репозитории.
*   **Генерация данных для интеграционных тестов.** Скрипты создают реалистичные коллекции с заданными параметрами.
*   **Анонимизация реальных данных.** При необходимости использования реальных коллекций все персональные данные удаляются или заменяются псевдонимами.
*   **Изоляция тестового окружения.** Каждый тестовый прогон использует временные директории и изолированную базу данных; после завершения окружение очищается.

**15.6.2 Конфигурация тестового окружения**

*   **Переопределение зависимостей.** В тестовом режиме внешние сервисы заменяются моками или тестовыми реализациями.
*   **Параметризация конфигурации.** Настройки приложения передаются как параметры, что позволяет тестировать разные сценарии без изменения кода.
*   **Логирование для отладки.** В тестовом режиме включается подробное логирование; логи сохраняются для анализа неудачных прогонов.

**15.7 Непрерывная интеграция и контроль качества**

Процессы сборки, тестирования и развёртывания автоматизированы для обеспечения скорости и надёжности доставки изменений.

**15.7.1 Конвейер непрерывной интеграции**

1.  **Триггеры.** Сборка запускается при каждом коммите в основную ветку и при создании запроса на слияние.
2.  **Этапы конвейера:**
    *   Установка зависимостей и кэширование артефактов.
    *   Статический анализ кода и проверка форматирования.
    *   Запуск модульных и виджет-тестов с расчётом покрытия.
    *   Сборка артефактов для разных конфигураций.
    *   Запуск интеграционных тестов на собранных артефактах.
    *   Проверка лицензий зависимостей и соответствия правовым требованиям.
3.  **Артефакты.** Успешно собранные версии сохраняются с метаданными для последующего развёртывания.

**15.7.2 Критерии принятия изменений**

*   Все автоматические проверки пройдены без ошибок.
*   Покрытие тестами не снизилось; новые функции покрыты соответствующими тестами.
*   Изменения соответствуют архитектурным принципам и не вводят избыточных зависимостей.
*   Документация обновлена, если изменения затрагивают публичные интерфейсы или пользовательский опыт.

**15.8 Жёсткие правила генерации кода (Тестирование и качество)**

Данные требования являются обязательными для всех последующих этапов реализации. Любое отклонение считается архитектурной ошибкой.

1.  **Тесты как часть определения готовности.** Функция не считается завершённой, пока не покрыта соответствующими тестами. Отсутствие тестов блокирует слияние кода.
2.  **Изоляция тестового окружения.** Все тесты должны выполняться в изолированном окружении; доступ к реальной файловой системе, сети или базе данных в тестах запрещён.
3.  **Детерминированность тестов.** Тесты не должны зависеть от порядка выполнения, времени выполнения или внешних состояний; каждый тест должен давать одинаковый результат при повторном запуске.
4.  **Автоматизация рутинных проверок.** Все повторяющиеся проверки выполняются автоматически в конвейере; ручные проверки допускаются только для сложных сценариев.
5.  **Регрессионная защита.** Каждое исправление ошибки сопровождается тестом, гарантирующим, что проблема не вернётся. Тесты на регрессию не удаляются без явного обоснования.
6.  **Производительность как требование.** Новые функции не должны ухудшать целевые метрики производительности; регрессия более десяти процентов требует оптимизации или пересмотра реализации.
7.  **Доступность как стандарт.** Все новые интерфейсные компоненты должны проходить проверки доступности; отсутствие поддержки вспомогательных технологий блокирует интеграцию кода.
8.  **Документирование сложных сценариев.** Тесты, проверяющие сложную логику или сценарии восстановления, должны сопровождаться комментариями, объясняющими проверяемое поведение и ожидаемый результат.

**16.2 Архитектура системы обновлений**

**16.2.1 Компоненты авто-апдейтера**

Система обновлений состоит из четырёх взаимосвязанных компонентов, каждый из которых выполняет строго определённую функцию:

**Компонент один: Проверка обновлений**

*   Запускается при старте приложения и периодически в фоновом режиме (интервал: двадцать четыре часа).
*   Запрашивает манифест последней версии с сервера распространения.
*   Сравнивает текущую версию приложения с доступной, учитывая канал распространения (стабильный, бета, разработческий).
*   Учитывает пользовательские настройки: отключение авто-проверки, выбор канала обновлений.

**Компонент два: Менеджер загрузки**

*   Скачивает установщик новой версии во временную директорию.
*   Проверяет целостность файла через сравнение контрольной суммы (алгоритм хеширования: эс-эйч-а-двести пятьдесят шесть).
*   Проверяет цифровую подпись установщика для подтверждения подлинности.
*   Не блокирует работу основного приложения: загрузка происходит в фоновом потоке с возможностью паузы и возобновления.

**Компонент три: Оркестратор установки**

*   Активируется при следующем запуске приложения или по явному запросу пользователя.
*   Корректно завершает работу текущей версии приложения.
*   Запускает установщик с параметрами тихой установки (без дополнительных диалогов).
*   Сохраняет пользовательские настройки и данные для переноса в новую версию.

**Компонент четыре: Страж отката**

*   Сохраняет предыдущую версию приложения в течение семи дней после успешного обновления.
*   Автоматически обнаруживает критические ошибки при первом запуске новой версии.
*   Выполняет откат к предыдущей стабильной версии при обнаружении проблем.
*   Логирует детали ошибки для последующего анализа разработчиками.

**16.2.2 Формат манифеста обновлений**

Манифест обновлений — это структурированный документ, содержащий метаданные о новой версии приложения. Он распространяется в формате, совместимом с машиночитаемой обработкой.

**Обязательные поля манифеста:**

*   Идентификатор версии в формате мажор.минор.патч.
*   Номер сборки для внутреннего отслеживания.
*   Дата и время публикации релиза.
*   Канал распространения: стабильный, бета, разработческий.
*   Ссылка на установщик для целевой платформы.
*   Контрольная сумма установщика для проверки целостности.
*   Размер установщика в байтах для предварительной оценки времени загрузки.
*   Цифровая подпись манифеста для подтверждения подлинности.
*   Список изменений в человекочитаемом формате.
*   Минимальная версия операционной системы для совместимости.
*   Флаг наличия критических изменений, требующих внимания пользователя.
*   Информация для отката: ссылка на предыдущую версию и её контрольная сумма.

**16.2.3 Проверка целостности и подлинности**

Процесс валидации обновлений включает два обязательных этапа:

**Этап один: Проверка контрольной суммы**

*   После завершения загрузки вычисляется контрольная сумма полученного файла.
*   Вычисленное значение сравнивается с ожидаемым значением из манифеста.
*   При несовпадении загрузка считается повреждённой, файл удаляется, пользователь получает уведомление с предложением повторить загрузку.

**Этап два: Проверка цифровой подписи**

*   Цифровая подпись установщика проверяется с использованием открытого ключа разработчика.
*   Проверка подтверждает, что установщик не был изменён после подписания и создан доверенным источником.
*   При неудачной проверке установка блокируется, пользователь получает предупреждение о потенциальной угрозе безопасности.

**16.5 Механизм авто-обновления**

**16.5.1 Жизненный цикл обновления**

Процесс обновления следует детерминированному сценарию, обеспечивающему предсказуемость и надёжность:

1.  **Инициация проверки**
    *   Приложение запрашивает информацию о доступных обновлениях.
    *   Сервер возвращает манифест с метаданными последней версии.
2.  **Анализ и уведомление**
    *   Приложение сравнивает версии и определяет необходимость обновления.
    *   Пользователь получает ненавязчивое уведомление о доступном обновлении.
3.  **Загрузка в фоне**
    *   При согласии пользователя начинается фоновая загрузка установщика.
    *   Прогресс загрузки отображается в строке состояния без прерывания работы.
4.  **Валидация и подготовка**
    *   После завершения загрузки выполняется проверка целостности и подлинности.
    *   Установщик подготавливается к запуску при следующем старте приложения.
5.  **Установка при перезапуске**
    *   При следующем запуске приложение инициирует процесс установки.
    *   Пользовательские данные и настройки сохраняются и переносятся в новую версию.
6.  **Завершение и отчёт**
    *   После успешной установки приложение запускается в новой версии.
    *   Пользователь получает краткий отчёт о завершённом обновлении.

*   Предлагает варианты действий: обновить сейчас, напомнить позже, пропустить версию.

**Индикатор прогресса**

*   Показывает процент завершения и предполагаемое время.
*   Предоставляет возможность приостановить или отменить загрузку.

**Экран завершения обновления**

*   Подтверждает успешную установку новой версии.
*   Предлагает ознакомиться с полным списком изменений.
*   Предоставляет доступ к настройкам для управления будущими обновлениями.

**17.4.3 Гибкость дорожной карты**

**Что может измениться:**

*   Сроки: сдвиг на две-четыре недели допустим при обнаружении критических багов или изменении приоритетов
*   Приоритеты: функции могут перемещаться между версиями по обратной связи пользователей и метрикам использования
*   Объём: незначительные фичи могут быть отложены ради стабильности и качества основных функций

**Что не меняется:**

*   Философия: «ноль трения, максимум контроля» остаётся неизменной во всех решениях
*   Качество: метрики производительности и доступности не могут быть снижены ради скорости разработки
*   Безопасность: шифрование, приватность, контроль данных — неприкосновенны и не подлежат компромиссам

...**17.6 Заключительное слово**

«Хорошая документация — это не описание того, что есть. Это инструкция к тому, что должно быть.»

Эта методичка — не просто список требований. Это **контракт между видением и реализацией**, между разработчиком и пользователем, между сегодняшним днём и целевой версией.

Каждый раздел, каждая таблица, каждый чек-лист — это инвестиция в то, чтобы приложение стало не просто «ещё одним инструментом», а **продолжением мысли пользователя**, которое чувствуется интуитивно, работает надёжно и масштабируется предсказуемо.

Когда код следует этой спецификации, а спецификация эволюционирует вместе с кодом через контролируемые версии — рождается продукт, которым хочется пользоваться, доверять и рекомендовать.

**Принцип реализации:** Если возникает выбор между «быстрее написать» и «правильнее описать» — выбирается второе. Ясность документации — это скорость разработки в долгосрочной перспективе.
