**📘 РАЗДЕЛ 1: ФУНДАМЕНТ, ДИЗАЙН-СИСТЕМА И ПРИНЦИПЫ UX/UI**

**1.1 Философия продукта и архитектурный мандат**

GlubLink проектируется как единое пространство для визуального
кураторства. Приложение отвергает традиционную файловую иерархию в
пользу динамических Workspaces. Контент существует в единственной
сущности «Блок», который может проявлять себя как медиа, заметка, ссылка
или коллекция. Пользователь не перемещает файлы --- он назначает
контекст. Приложение исполняет правила, а не требует ручного
обслуживания.

**Жёсткие мандаты:**

1.  **Zero-Friction UX.** Любое действие требует не более двух кликов.
    Сложные операции скрыты за контекстными меню или выполняются
    автоматически.

2.  **Non-Blocking Architecture.** Интерфейс никогда не замораживается.
    Все тяжёлые процессы (сканирование, индексация, ИИ-анализ, экспорт)
    выполняются в фоновых потоках с мгновенным возвратом управления.

3.  **Reversible State.** Каждое изменение пользовательского контента
    обратимо. Корзина хранит удалённое 24 часа. Фильтры сбрасываются
    одним кликом. Конфигурации Workspaces не меняют физические файлы без
    явного подтверждения.

4.  **Context-Aware UI.** Интерфейс адаптируется под текущий фокус.
    Панели появляются только при необходимости. Навигация предсказуема.
    Анимации следуют за намерением пользователя, а не за таймерами.

**1.2 Дизайн-система: токены и визуальный язык**

Визуальный язык строго детерминирован. Использование значений вне
указанных токенов запрещено. Все цвета, размеры и отступы масштабируются
пропорционально системному DPI.

**Цветовая палитра:**

-   Акцент (градиент): #6C63FF → #FF6B9D

-   Фон (Тёмная тема): #121212

-   Фон (Светлая тема): #F5F5F7

-   Поверхность (Тёмная): #1E1E1E

-   Поверхность (Светлая): #FFFFFF

-   Текст основной (Тёмная/Светлая): #EAEAEA / #1A1A1A

-   Текст вторичный: #9E9E9E / #6B6B6B

-   Ошибка: #FF5252

-   Успех: #4CAF50

-   Предупреждение: #FFC107

-   Категории тегов: Artist #FF6B9D, Copyright #9D6BFF, Character
    #6B9DFF, Species #6BFF9D, General #FFD96B, Meta #9D9D9D, References
    #FF9D6B.

**Типографика:**

-   Семейство: Системные шрифты платформы (Segoe UI, SF Pro, Inter).

-   Размеры: 11, 12, 13, 14, 16, 20 (заголовки), 24 (экраны).

-   Жирность: 400 (тело), 500 (интерактив), 600 (заголовки), 700
    (акценты).

-   Межстрочный интервал: 1.4 для текста, 1.2 для заголовков.

-   Масштабирование: поддерживается системный масштаб до 200%. Интерфейс
    сохраняет пропорции и не ломает вёрстку.

**Геометрия и отступы:**

-   Сетка отступов: кратна 4. Допустимые значения: 4, 8, 12, 16, 24, 32,
    48.

-   Скругления: Кнопки 8px, Карточки 12px, Модальные окна 16px,
    Чипы/Поля ввода 50% (капсула), Аватары/Иконки 12px.

-   Тени: Тёмная тема --- 0 2px 8px rgba(0,0,0,0.4). Светлая тема --- 0
    2px 12px rgba(0,0,0,0.08). Только для парящих элементов.

**1.3 Физика анимаций и микро-взаимодействия**

Все переходы используют пружинную физику. Запрещено использование
линейных таймеров или фиксированных кривых ease-in-out для интерактивных
элементов. Анимации должны ощущаться как продолжение жеста пользователя.

**Конфигурации пружин (Stiffness / Damping / Mass):**

-   Микровзаимодействия (наведение, нажатие кнопок, чипов): 280 / 22 /
    1.0. Длительность: 100--150 мс.

-   Боковые панели и Drawer: 200 / 18 / 1.0. Длительность: 200--300 мс.

-   Навигация между экранами (Hero-переходы, смена Workspace): 150 / 15
    / 1.2. Длительность: 300--400 мс.

-   Появление сетки контента (каскад): 250 / 20 / 1.0. Задержка между
    элементами: 30 мс.

-   Удаление/архивирование: 180 / 16 / 1.0. Направлено в сторону
    корзины. Не прерывается после подтверждения.

**Принципы интерактивности:**

1.  **Прерываемость.** Любая анимация, кроме операций удаления и
    системных алертов, мгновенно реагирует на противоположный жест или
    наведение.

2.  **Визуальный вес.** Элементы не «прыгают». Масштабирование при
    наведении не превышает 1.02. Тень усиливается пропорционально
    подъёму.

3.  **Тактильный отклик.** Курсор меняется на SystemMouseCursors.click
    только на активных зонах. Запрещено применять его к статическому
    тексту или иконкам-плейсхолдерам.

4.  **Состояния загрузки.** Запрещены спиннеры в контентных областях.
    Используются пульсирующие скелетоны с точной геометрией будущего
    контента. Прогресс-бары применяются только для операций \>2 секунд.

**1.4 Принципы премиум-UX: нулевое трение**

Интерфейс проектируется вокруг интуитивных паттернов. Обучение
пользователя происходит через действие, а не через инструкции.

**Правила навигации и фокуса:**

-   Главный экран содержит только три зоны: Навигация (Workspaces),
    Контентная область, Контекстная панель.

-   Переход в Workspace мгновенный. Фильтры применяются в реальном
    времени без перезагрузки вида.

-   Поиск универсален. Поддерживает позитивные теги, негативные (префикс
    -), логическое ИЛИ (\|\|), группировку (). Результат парсится в
    визуальные чипы немедленно.

-   Пустые состояния содержат только одно целевое действие (например,
    «Выбрать папку»). Запрещены декоративные иллюстрации, не несущие
    функциональной нагрузки.

**Обработка состояний:**

-   **Офлайн.** Приложение работает полностью автономно. Индикатор
    статуса размещается в строке состояния. ИИ-функции переходят в режим
    кэша без блокировки интерфейса.

-   **Ошибки.** Сообщения содержат причину и конкретное решение.
    Запрещены формулировки «Произошла ошибка» или «Попробуйте позже».
    Каждое предупреждение предлагает кнопку действия или отмену.

-   **Многопоточность.** UI-поток обрабатывает только отрисовку и ввод.
    Все операции с файловой системой, базой данных и сетью вынесены в
    изолированные потоки. Передача данных происходит только через
    сериализуемые примитивы.

**1.5 Архитектура интерфейса и компоновка экранов**

Структура приложения строится по модульному принципу. Каждый экран
является композицией независимых панелей, управляемых единым роутером.

**Макет рабочего пространства:**

1.  **Левая панель (Навигация).** Фиксированная ширина в развёрнутом
    состоянии. Содержит быстрые фильтры, список Workspace, настройки. В
    свёрнутом состоянии отображает только иконки. Переход между
    состояниями анимирован.

2.  **Центральная область (Контент).** Занимает всё оставшееся
    пространство. Содержит сетку медиа, заметок или дашборд.
    Адаптируется под размер окна. Виртуализация включена по умолчанию
    для любых списков \>50 элементов.

3.  **Правая панель (Контекст).** Выезжает справа. Содержит детали
    выбранного элемента, теги, свойства, связи. Автоматически скрывается
    при клике вне области или при переключении фокуса. Ширина
    фиксирована, содержимое скроллится независимо от основного контента.

**Поведение сетки контента:**

-   Поддерживает четыре режима: Masonry (кирпичная кладка), Justified
    (выровненные ряды), List (детальный список), Timeline (хронология).

-   Переключение режима мгновенно пересчитывает layout без потери
    позиции скролла.

-   Карточки отображают превью, тип контента и базовые мета-теги. При
    наведении плавно увеличиваются и показывают панель быстрых действий
    (тег, избранное, открыть).

-   Пагинация ленивая. Загрузка следующей порции элементов происходит
    при достижении порога прокрутки. Предзагрузка буфера: +2 экрана
    вперёд.

**1.6 Жёсткие правила генерации кода (UI/UX слой)**

Данные требования являются обязательными для всех последующих этапов
реализации. Любое отклонение считается архитектурной ошибкой.

1.  **Единый источник стилей.** Все цвета, размеры, шрифты и радиусы
    берутся исключительно из глобального объекта темы. Хардкод значений
    в виджетах запрещён.

2.  **Состояния виджетов.** Каждый интерактивный элемент обязан иметь
    состояния: default, hover, focus, pressed, disabled. Переходы между
    состояниями анимируются через пружинные кривые.

3.  **Доступность.** Все кнопки, поля ввода и интерактивные иконки
    содержат семантические метки. Контрастность текста соответствует
    стандарту 4.5:1. Навигация с клавиатуры покрывает 100% интерактивных
    зон.

4.  **Отказ от глобального состояния в UI.** Виджеты не хранят
    бизнес-данные. Они получают состояние через реактивные провайдеры.
    Изменение данных триггерит перерисовку только затронутых
    поддеревьев.

5.  **Обработка ошибок рендеринга.** Все внешние изображения и
    медиа-превью оборачиваются в обработчики ошибок. При невозможности
    декодирования отображается стандартизированный плейсхолдер с иконкой
    типа контента. Блокировка потока недопустима.

**📘 РАЗДЕЛ 2: АРХИТЕКТУРА ПРИЛОЖЕНИЯ, МОДЕЛЬ ДАННЫХ И НАВИГАЦИЯ**

**2.1 Архитектурный мандат и слоистая модель**

Приложение строится по принципам Clean Architecture в комбинации с
Feature-First организацией. Это означает, что каждый функциональный
модуль содержит собственные слои данных, домена и представления.
Зависимости строго однонаправлены: внешние слои зависят от внутренних,
обратные связи запрещены.

**Слоистая структура:**

1.  **Presentation (UI/UX).** Содержит виджеты, обработчики жестов,
    локальное состояние интерфейса и привязки к провайдерам. Запрещено
    хранить бизнес-логику, прямые запросы к БД или файловую систему.

2.  **Domain (Ядро).** Содержит интерфейсы репозиториев, чистые функции
    валидации, правила фильтрации Workspaces, модели сущностей и
    абстрактные описания операций. Не зависит от фреймворков, БД или
    сети.

3.  **Data (Инфраструктура).** Содержит реализации репозиториев,
    адаптеры к Isar, работу с изолятами, кэширование, парсинг файлов и
    обработку сетевых ответов. Отвечает за преобразование сырых данных в
    доменные модели.

**Правила модульности:**

-   Каждый feature-модуль (Дашборд, Галерея, Медиа-просмотр, Заметки,
    Настройки, Workspaces) инкапсулирует свою логику.

-   Пересечение данных между модулями происходит только через доменные
    интерфейсы.

-   Глобальные сервисы (Синхронизация, ИИ-пайплайн, Логирование,
    Настройки темы) вынесены в отдельный слой Core и инжектируются как
    абстракции.

-   Запрещено использовать глобальные синглтоны для хранения состояния.
    Только реактивные провайдеры.

**2.2 Единая сущность «Контент-блок»**

Вместо разрозненных таблиц для изображений, заметок и ссылок
используется единая модель. Тип блока определяет лишь набор
дополнительных полей и способ рендеринга. Все операции поиска,
тегирования и привязки к Workspace применяются одинаково.

**Обязательные поля любого блока:**

-   Уникальный идентификатор (UUID)

-   Тип сущности (Медиа / Заметка / Ссылка / Коллекция)

-   Дата создания и дата добавления в приложение

-   Список привязанных тегов (массив строковых идентификаторов)

-   Список связей с другими блоками (пары: ID целевого блока + тип
    связи)

-   Флаги состояния (Избранное, Архив, Корзина, Статус доступа)

-   Мета-данные безопасности (хеш содержимого, флаг целостности,
    путь-источник)

**Специфика по типам:**

-   **Медиа:** физический путь, разрешение, длительность, формат, путь к
    кэшированному превью, ориентация.

-   **Заметка:** текстовое содержимое в совместимом формате, список
    вложенных медиа-блоков, счётчик слов, дата последнего
    редактирования.

-   **Ссылка:** URL, заголовок страницы, описание, путь к
    превью-фавикону, статус доступности ресурса.

-   **Коллекция:** не хранит контент, содержит только правила
    фильтрации, порядок сортировки и список вручную закреплённых
    элементов.

**Правила жизненного цикла блока:**

1.  Создание атомарно. Если операция прервана, данные не остаются в
    частичном состоянии.

2.  Удаление всегда мягкое. Запись помечается как перемещённая в
    корзину. Физическое удаление происходит только после истечения срока
    хранения или явного подтверждения.

3.  Изменение метаданных не требует перезаписи файла на диске.
    Обновляется только индекс в базе данных.

4.  Все блоки индексируются для полнотекстового и тегового поиска при
    создании. Индекс обновляется реактивно.

**2.3 Система Workspaces: Виртуальные коллекции**

Workspace --- это не папка, а динамическое правило выборки. Приложение
не перемещает файлы, оно вычисляет принадлежность на лету или кэширует
результат выборки.

**Архитектура правила Workspace:**

-   **Фильтры включений:** теги, типы контента, диапазоны дат,
    минимальное/максимальное разрешение, наличие заметок.

-   **Фильтры исключений:** негативные теги, скрытые форматы, архивные
    блоки, контент с определёнными мета-флагами.

-   **Логика объединения:** ИЛИ для групп тегов, И для пересечения
    категорий, приоритет ручных закреплений над автоматическими
    правилами.

-   **Сортировка и группировка:** настраивается пользователем на уровне
    каждого Workspace (по дате, по популярности тегов, по визуальной
    схожести, хронология).

**Поведение и синхронизация состояния:**

1.  Создание или редактирование правила мгновенно пересчитывает виджет
    галереи без перезагрузки экрана.

2.  При изменении тега у блока все затронутые Workspace обновляются
    реактивно.

3.  Конфигурация каждого Workspace (режим сетки, закреплённые чипы,
    позиция скролла, активные фильтры) сохраняется локально.

4.  Виртуальный режим (по умолчанию): файлы остаются на оригинальных
    путях. Приложение хранит маппинг Workspace → ID блоков.

5.  Физический режим (опционально): создаются жёсткие ссылки в
    структурированных директориях. Логика приложения не меняется,
    различие только в способе доступа к файловой системе.

**Правила принадлежности:**

-   Блок может принадлежать любому количеству Workspace одновременно.

-   Удаление блока из одного Workspace не удаляет его из других и не
    затрагивает оригинальный файл.

-   Дубликаты файлов предотвращаются на уровне уникального хеша
    содержимого, а не пути.

**2.4 Управление состоянием и реактивность**

Состояние приложения управляется через единую реактивную шину. Данные
текут однонаправленно: Источник состояния → Преобразование →
Представление.

**Иерархия провайдеров:**

1.  **Базовые:** конфигурация приложения, подключение к локальной базе
    данных, сетевой клиент, менеджер изолятов, хранилище настроек
    пользователя.

2.  **Репозитории:** абстрактные интерфейсы для операций CRUD, поиска,
    фильтрации, тегирования и синхронизации.

3.  **Фичевые:** состояния Workspaces, пулы тегов, активные фильтры,
    статус сканирования, история навигации.

4.  **UI-представления:** локальное состояние виджетов (открыта ли
    панель, выбранный элемент сетки, анимация загрузки). Не содержат
    бизнес-данных.

**Обработка асинхронных потоков:**

-   Все состояния загрузки, успеха и ошибки описываются единым
    типом-обёрткой.

-   Виджеты реагируют на изменения состояния автоматически. Нет ручных
    вызовов setState для данных.

-   Ошибки не пробрасываются в UI как исключения. Они преобразуются в
    описательные объекты состояния с инструкциями по восстановлению.

-   Кэширование результатов частых запросов (топ-теги, недавние блоки,
    активный Workspace) хранится в памяти и инвалидируется при изменении
    источника.

**Требования к изолятам:**

-   Тяжёлые операции (рекурсивное сканирование директорий, генерация
    хешей, кодирование превью, пакетная запись в БД) выполняются в
    отдельных потоках.

-   Передача данных между потоками происходит только через сериализуемые
    примитивы (строки, числа, булевы значения, списки). Передача сложных
    объектов запрещена.

-   UI-поток никогда не блокируется ожиданием. Прогресс передаётся
    порционно, интерфейс обновляется плавно.

**2.5 Маршрутизация и контроль навигации**

Навигация строится на едином декларативном роутере. История переходов
сохраняется, поддерживает возврат и вперёд, не теряет контекст фильтров
при переключении экранов.

**Структура маршрутов:**

-   Главная панель (Дашборд): стартовая точка, отображает рекомендации,
    быстрые действия и сводку по коллекции.

-   Контекст Workspace: путь включает идентификатор коллекции. Сохраняет
    активные фильтры и режим отображения в параметрах маршрута.

-   Полноэкранный просмотр медиа: отдельный маршрут с передачей
    идентификатора блока и индекса в текущей выборке. Поддерживает
    возврат в родительский контекст без потери позиции скролла.

-   Редактор заметок: изолированный маршрут для работы с текстом и
    вложениями. Автосохранение состояния при сворачивании или
    переключении.

-   Настройки: модальный или стековый маршрут, не сбрасывающий основной
    контекст приложения.

**Поведение истории:**

-   Каждый значимый переход (смена Workspace, открытие медиа, применение
    сложного фильтра) создаёт запись в стеке.

-   Навигация назад возвращает к предыдущему логическому состоянию, а не
    просто к предыдущему экрану.

-   При восстановлении сессии (перезапуск приложения) роутер
    автоматически направляет пользователя в последний активный контекст
    с восстановленными параметрами.

-   Глубокие ссылки (deep links) обрабатываются через парсинг внешних
    параметров и трансляцию их во внутренние состояния фильтров.

**Модальность и оверлеи:**

-   Быстрые действия (добавление тега, закрепление, экспорт) открываются
    как оверлеи поверх текущего контекста. Закрытие не разрушает
    состояние.

-   Диалоги подтверждения (удаление, смена режима организации, очистка
    кэша) блокируют взаимодействие с фоном до принятия решения.

-   Анимации появления и скрытия маршрутов синхронизированы с пружинной
    физикой, описанной в Разделе 1. Прерывание жеста отменяет переход
    без артефактов.

**2.6 Жёсткие правила генерации кода (Архитектура)**

Данные требования являются обязательными для всех последующих этапов
реализации. Любое отклонение считается архитектурной ошибкой.

1.  **Запрет бизнес-логики в представлении.** Виджеты только
    отрисовывают данные и передают события. Все вычисления, фильтрация и
    валидация происходят в слое домена или репозитория.

2.  **Строгая типизация моделей.** Запрещено использование динамических
    типов в доменных сущностях. Все перечисления, статусы и категории
    описываются явно через замкнутые типы.

3.  **Атомарность операций БД.** Любое действие, затрагивающее более
    одной записи или несколько таблиц (теги + связи + индекс),
    выполняется в транзакции. При ошибке происходит полный откат.

4.  **Изоляция побочных эффектов.** Запросы к файловой системе, сети или
    внешним API инкапсулированы в репозиториях. Презентационный слой не
    импортирует пакеты ввода-вывода.

5.  **Обработка ошибок на границе фичи.** Каждый модуль имеет
    собственный обработчик исключений. Ошибки логируются, преобразуются
    в пользовательские сообщения и не приводят к падению приложения.

6.  **Реактивная инвалидация кэша.** При изменении данных источник
    состояния автоматически уведомляет всех подписчиков. Ручное
    обновление UI запрещено.

7.  **Детерминированная навигация.** Роутер является единственным
    источником правды о текущем экране. Переходы между состояниями не
    дублируются через push/pop без явной необходимости.

8.  **Тестируемость слоёв.** Каждый репозиторий и доменная функция
    должны быть изолированы от UI. Это позволяет запускать юнит- и
    интеграционные тесты без рендеринга интерфейса.
