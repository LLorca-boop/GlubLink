**📘 РАЗДЕЛ 7: СИСТЕМА ИНТЕЛЛЕКТУАЛЬНОГО ТЕГИРОВАНИЯ И
АВТО-ОРГАНИЗАЦИИ**

**7.1 Философия системы: «Умная простота, полный контроль»**

Система тегирования в GlubLink --- это не просто метки, а семантический
слой, который превращает коллекцию из набора файлов в структурированную
базу знаний. Приложение не заставляет пользователя вручную размечать
каждый файл --- оно учится на действиях, предлагает релевантные теги и
исполняет правила организации.

**Жёсткие мандаты:**

1.  **Автономность с контролем.** Любое автоматическое действие
    (добавление тега, перемещение в коллекцию) должно быть обратимым,
    прозрачным и обучающимся на коррекциях пользователя.

2.  **Трёхуровневая архитектура тегов.** Система работает офлайн с
    локальным ядром, расширяется кэшем пользователя и при необходимости
    обращается к внешним источникам.

3.  **Контекстная релевантность.** Теги отображаются и предлагаются
    только в соответствии с текущей выборкой, историей действий и
    профилем коллекции.

4.  **Нулевое трение при ручном тегировании.** Добавление, удаление и
    редактирование тегов выполняется не более чем за два действия, с
    мгновенной визуальной обратной связью.

**7.2 Архитектура тегов: Трёхуровневая модель**

Система тегов строится по иерархическому принципу, где каждый уровень
имеет чёткую зону ответственности, стратегию кэширования и условия
активации.

**Уровень 1: Core Tags (Локальное ядро)**

-   **Состав:** \~5 000 наиболее популярных тегов из эталонных
    источников (включая все мета-теги: image, gif, video, highres).

-   **Хранение:** Полностью в локальной базе Isar, без зависимости от
    сети.

-   **Обновление:** Опциональная синхронизация раз в 30 дней через
    фоновый процесс.

-   **Использование:** Базовый пул для автодополнения, валидации и
    первичного авто-тегирования.

-   **Защита:** Мета-теги помечены флагом isProtected --- не могут быть
    удалены или отредактированы пользователем.

**Уровень 2: User Cache (Персональный кэш)**

-   **Состав:** Все теги, которые пользователь когда-либо использовал,
    подтвердил или создал вручную.

-   **Хранение:** В Isar с привязкой к идентификатору устройства и
    временными метками.

-   **TTL:** 90 дней неактивности → автоматическая архивация (не
    удаление).

-   **Расширение:** Автоматически пополняется при подтверждении
    предложений ИИ или ручном добавлении.

-   **Приоритет:** В автодополнении теги из User Cache отображаются
    первыми, независимо от глобальной популярности.

**Уровень 3: Dynamic Sources (Внешние источники)**

-   **Источники:** e621 API (поиск по MD5), SauceNAO (reverse image
    search), Qwen API (перевод и семантический анализ).

-   **Активация:** Только при отсутствии совпадения в Core Tags и User
    Cache, и только с явного согласия пользователя.

-   **Rate Limiting:** Строгое соблюдение лимитов (60 запросов/мин для
    e621, 10/мин для Qwen).

-   **Кэширование:** Результаты внешних запросов сохраняются в User
    Cache с пометкой источника и сроком жизни.

-   **Fallback:** При недоступности сети система автоматически
    переключается на офлайн-режим без блокировки интерфейса.

**7.3 Категории тегов и визуальная система**

Каждый тег принадлежит к одной из семи предопределённых категорий.
Категория определяет цвет, порядок отображения, приоритет в поиске и

**7.4 Интеллектуальное авто-тегирование: Пайплайн обработки**

Процесс авто-тегирования нового медиафайла представляет собой
детерминированный пайплайн с чёткими порогами уверенности и стратегиями
эскалации.

**Этап 1: Локальный анализ**

-   Вычисление SHA-256 хеша файла для проверки дубликатов.

-   Извлечение базовых метаданных: разрешение, длительность, формат,
    EXIF (если доступно).

-   Быстрая классификация типа: image / gif / video → автоматическое
    добавление соответствующего мета-тега.

**Этап 2: Визуальный эмбеддинг**

-   Генерация векторного представления (512 измерений) через локальную
    модель CLIP.

-   Сравнение с базой эмбеддингов уже обработанных файлов в коллекции.

-   При совпадении \>90% → предложение применить теги из похожего файла.

**Этап 3: Определение профиля коллекции**

-   Анализ распределения тегов в текущей коллекции:

    -   Если \>60% тегов относятся к категории Species → активируется
        «Фурри-профиль» (приоритет распознавания персонажей).

    -   Если \>50% тегов --- Artist → «Арт-профиль» (приоритет поиска
        автора).

    -   Иначе → «Общий профиль» (баланс всех категорий).

-   Профиль влияет на веса алгоритма и порядок предложений.

**Этап 4: Классификация по порогу уверенности**

-   **Confidence \>85%:** Тег применяется автоматически, записывается в
    лог с пометкой auto-applied.

-   **Confidence 70--85%:** Тег добавляется в список «Предложенные»
    (отображается серым чипом с иконкой ✨).

-   **Confidence \<70%:** Тег игнорируется, не загружает интерфейс
    пользователя.

-   Пользователь может в любой момент просмотреть и
    подтвердить/отклонить предложенные теги.

**Этап 5: Интернет-фоллбэк (опционально)**

-   Если локальные методы не дали результата с достаточной уверенностью:

    1.  Поиск по MD5 на e621 API (мгновенно, 100% точность при
        совпадении).

    2.  Если не найдено → reverse search через SauceNAO.

    3.  Если не найдено → предложение пользователю: «Новый персонаж?
        Дайте имя».

-   Все внешние запросы выполняются в фоне, без блокировки UI.

**Этап 6: Сохранение и обучение**

-   Запись результата в Isar: связь медиа → теги + метаданные
    уверенности.

-   Обновление весов визуальных паттернов: если пользователь подтвердил
    тег → усиление связи «эмбеддинг → тег».

-   Кэширование результата: хеш файла → список тегов, TTL 90 дней.

**7.5 Механизм «Character Identity»: Распознавание персонажей**

Проблема: Один и тот же персонаж в разных позах, стилях или освещении
может восприниматься алгоритмом как разные сущности.

**Решение:** Создание сущности «Идентификатор персонажа», которая
объединяет множество визуальных эмбеддингов в один семантический
кластер.

**Структура Character Identity:**

-   id: Уникальный UUID.

-   name: Имя персонажа (задано пользователем или найдено во внешних
    источниках).

-   visualCluster: Массив из 10--50 эталонных эмбеддингов,
    представляющих персонажа.

-   coreTags: Неизменные признаки (например, fox, anthro, blue_eyes).

-   variableTags: Изменяемые признаки (например, standing, night,
    smiling).

-   adaptiveThreshold: Динамический порог уверенности (0.75--0.90),
    адаптирующийся под частоту подтверждений пользователя.

-   confirmedByUser: Флаг, указывающий, что кластер был явно
    подтверждён.

**Алгоритм распознавания:**

1.  Новый арт → генерация эмбеддинга.

2.  Поиск ближайших соседей в локальной базе по косинусному сходству.

3.  Если совпадение \>85% с известным Character Identity:

    -   Предложение: «Похоже на персонажа \'Zara\'. Применить теги?»

    -   При подтверждении: добавление нового эмбеддинга в кластер для
        улучшения будущих распознаваний.

4.  Если совпадение 70--85%:

    -   Адаптивное снижение порога (если пользователь часто подтверждает
        похожие арты).

    -   Предложение с пометкой: «Возможно, это Zara (другое освещение)».

5.  Если \<70%:

    -   Поиск во внешних источниках по визуальному сходству.

    -   Если не найдено → запрос имени у пользователя.

**Обучение на коррекциях:**

-   При удалении пользователем автоматически добавленного тега:

    -   Уменьшение веса связи «визуальный паттерн → тег» на коэффициент
        0.2.

-   При ручном добавлении тега, который не был предложен:

    -   Увеличение веса связи на коэффициент 0.15.

-   Если добавленный тег относится к категории Character:

    -   Обновление визуального кластера соответствующего Character
        Identity.

**7.6 Визуальный билдер запросов и умный поиск**

Поисковая строка --- это не просто текстовое поле, а парсер логических
выражений, который мгновенно преобразует ввод в визуальные токены и
передаёт структурированный запрос в ядро выборки.

**Логика парсинга и визуализации:**

-   **Позитивные теги:** Ввод fox → отображается синий чип \[fox\].

-   **Негативные теги:** Ввод -sketch → отображается красный зачёркнутый
    чип \[-sketch\].

-   **Логическое ИЛИ:** Ввод (zara \|\| rex) → отображается фиолетовый
    групповой чип \[(zara \|\| rex)\].

-   **Вложенность и скобки:** Парсер поддерживает приоритет операций.
    Ошибки синтаксиса подсвечиваются мягким предупреждением без
    блокировки ввода.

-   **Мгновенная конвертация:** Каждый символ ввода триггерит пересчёт
    токенов. Запрос применяется к выборке в реальном времени (дебаунс
    150 мс для оптимизации).

**Автодополнение и контекстные подсказки:**

-   При вводе ≥2 символов открывается оверлей с предложениями.

-   Приоритет выдачи: теги из User Cache → теги из Core Tags → теги из
    текущей выборки → предложение поиска в API.

-   Каждый вариант отображает иконку категории, название и счётчик
    использования в коллекции.

-   Выбор варианта клавиатурой (стрелки + Enter) или кликом мгновенно
    формирует чип и обновляет запрос.

**Функция «✨ Авто-настройка»:**

-   Кнопка доступна в настройках вида. Анализирует текущую выборку и
    предлагает готовые комбинации фильтров на основе частоты тегов,
    визуального сходства или метаданных.

-   Предложение отображается как модальная подсказка над панелью.
    Кнопки: «Применить», «Игнорировать», «Показать логику выбора».

-   Действие не изменяет правила Workspace, только текущий сеанс
    фильтрации.

**7.7 Пользовательский интерфейс тегирования: Нулевое трение**

Интерфейс работы с тегами спроектирован так, чтобы минимизировать
количество действий и максимизировать обратную связь.

**Правая панель галереи: Контекстные теги**

-   Отображаются ТОЛЬКО теги, присутствующие в текущей видимой выборке
    или у выбранного блока.

-   Сортировка по умолчанию: по количеству использований (убывание) → по
    алфавиту → по категории.

-   Группировка по категориям с возможностью скрыть/показать категорию
    через чекбокс.

-   Клик по названию тега → мгновенная фильтрация галереи по этому тегу.

-   Клик по «+» рядом с тегом → добавление тега в поисковую строку как
    чип.

-   Визуальные индикаторы: цветной квадрат категории + название +
    счётчик.

**Взаимодействие с тегами в полноэкранном режиме:**

-   Список применённых тегов с цветовой индикацией категорий.

-   Кнопка \[+\] открывает поле быстрого поиска с автодополнением из
    пула тегов.

-   Клик по тегу мгновенно применяет его к файлу и закрывает поле ввода.

-   Кнопка \[×\] удаляет тег. Операция обратима через Ctrl+Z в пределах
    сессии.

-   Изменение тегов реактивно обновляет пул галереи и индексы без
    перезагрузки.

**Менеджер тегов (модальный диалог):**

-   Доступен через кнопку «Управление всеми тегами» в правой панели.

-   Список всех тегов с фильтрацией по названию и категории.

-   Возможность создания нового тега: ввод названия + выбор категории.

-   Редактирование: изменение названия, категории, описания.

-   Удаление: только для незащищённых тегов, с подтверждением и
    предупреждением о влиянии на связанные блоки.

-   Все изменения применяются мгновенно и реактивно распространяются на
    все представления.

**7.9 Жёсткие правила генерации кода (Система тегирования)**

Данные требования являются обязательными для всех последующих этапов
реализации. Любое отклонение считается архитектурной ошибкой.

1.  **Единый источник истины для тегов.** Все операции с тегами
    (добавление, удаление, редактирование) выполняются только через
    реактивный провайдер. Запрещено дублировать логику управления тегами
    в презентационном слое.

2.  **Строгая типизация категорий.** Категория тега описывается через
    enum TagCategory с явными значениями. Запрещено использовать
    строковые или числовые литералы для сравнения категорий.

3.  **Атомарность операций с тегами.** Любое изменение, затрагивающее
    связь «медиа → теги», выполняется в транзакции. При ошибке
    происходит полный откат, и пользователь получает понятное сообщение.

4.  **Изоляция побочных эффектов.** Запросы к внешним API (e621, Qwen,
    SauceNAO) инкапсулированы в репозиториях. Презентационный слой не
    импортирует пакеты сетевого взаимодействия.

5.  **Реактивное обновление интерфейса.** Изменение тега у одного блока
    мгновенно уведомляет все подписанные виджеты (галерея, правая
    панель, фуллскрин). Запрещено ручное обновление через setState для
    данных.

6.  **Кэширование с чёткими стратегиями.** Результаты авто-тегирования,
    внешние запросы и эмбеддинги кэшируются с указанием TTL и стратегии
    эвикции (LRU). Прямое чтение без кэша запрещено.

7.  **Обработка ошибок на границе фичи.** Каждый модуль системы
    тегирования имеет собственный обработчик исключений. Ошибки
    логируются, преобразуются в пользовательские сообщения и не приводят
    к падению приложения.

8.  **Доступность интерфейса тегирования.** Все чипы, поля ввода и
    кнопки управления тегами содержат семантические метки, поддерживают
    навигацию с клавиатуры и имеют достаточную контрастность.

**📘 РАЗДЕЛ 8: СИНХРОНИЗАЦИЯ, РЕЗЕРВНОЕ КОПИРОВАНИЕ И УПРАВЛЕНИЕ
ДАННЫМИ**

**8.1 Философия синхронизации: «Офлайн-первый, облако-второй»**

Система управления данными в GlubLink проектируется с приоритетом
локальной работы. Пользователь получает полный доступ ко всем функциям
приложения без подключения к интернету. Синхронизация с облаком --- это
фоновый, необязательный процесс, который расширяет возможности, но не
является требованием для базовой функциональности.

**Жёсткие мандаты:**

1.  **Локальная автономность.** Все операции с контентом (добавление,
    редактирование, тегирование, удаление) выполняются мгновенно на
    локальной базе данных. Сетевые задержки не влияют на отзывчивость
    интерфейса.

2.  **Невидимая синхронизация.** Процесс обмена данными между
    устройствами происходит в фоне, без блокировки пользовательского
    потока. Пользователь уведомляется о статусе только при критических
    событиях.

3.  **Конфликтоустойчивость.** Любое расхождение в данных между
    устройствами разрешается автоматически по предсказуемым правилам.
    Ручное вмешательство требуется только в исключительных случаях.

4.  **Обратимость изменений.** Любое действие, синхронизированное между
    устройствами, может быть отменено в пределах настраиваемого
    временного окна. История изменений хранится локально и
    синхронизируется по запросу.

**8.2 Архитектура данных: Трёхуровневая модель**

Система хранения и синхронизации данных строится по иерархическому
принципу, где каждый уровень имеет чёткую зону ответственности и
стратегию репликации.

**Уровень 1: Локальная база данных (Источник истины)**

-   **Технология:** Isar --- высокоскоростная локальная БД с поддержкой
    индексов и транзакций.

-   **Содержимое:** Полная копия всех пользовательских данных:
    контент-блоки, теги, связи, настройки, история изменений.

-   **Доступ:** Мгновенный, без сетевых задержек. Все операции
    чтения/записи выполняются в изолированных потоках.

-   **Резервирование:** Автоматическое создание локальных бэкапов каждые
    4 часа с хранением последних 7 версий.

**Уровень 2: Очередь синхронизации (Буфер изменений)**

-   **Назначение:** Хранение локальных изменений, ожидающих отправки в
    облако или получения с других устройств.

-   **Структура записи:**

    -   Уникальный идентификатор изменения (UUID)

    -   Тип сущности и идентификатор затронутого блока

    -   Тип операции (создание, обновление, удаление)

    -   Векторные часы для определения причинно-следственных связей

    -   Хеш содержимого для проверки целостности

    -   Приоритет синхронизации (критический, высокий, средний, низкий)

-   **Стратегия отправки:**

    -   Критические изменения (удаление, права доступа) --- немедленная
        попытка отправки

    -   Высокий приоритет (теги, заметки) --- отправка в течение 30
        секунд

    -   Средний/низкий приоритет (статистика, кэш) --- отправка при
        стабильном соединении

-   **Обработка ошибок:** Экспоненциальная задержка между повторными
    попытками (5с → 15с → 60с → 300с). После 5 неудач --- отложение до
    следующего запуска.

**Уровень 3: Облачный синк-сервер (Координатор)**

-   **Поддерживаемые провайдеры:** Google Drive, Dropbox, самохостинг
    через WebDAV.

-   **Формат данных:** Только дельта-изменения, а не полные копии.
    Каждый пакет содержит минимальный набор данных для применения
    изменения.

-   **Шифрование:** Все данные шифруются на устройстве-отправителе перед
    передачей. Ключи шифрования никогда не покидают устройство
    пользователя.

-   **Вебхуки:** Опциональная интеграция с Telegram-ботом для
    уведомлений о статусе синхронизации и критических событиях.

**8.3 Версионирование и разрешение конфликтов**

Для обеспечения согласованности данных между устройствами без
центрального сервера используется механизм векторных часов (Vector
Clocks).

**Правила работы с векторными часами:**

1.  При создании новой записи: version = { current_device_id: 1 }

2.  При локальном изменении: version\[current_device_id\]++

3.  При получении удалённого изменения:

    -   Если версия новее (все таймстемпы ≥ локальных) → применить
        изменение

    -   Если версия старее (все таймстемпы ≤ локальных) → игнорировать

    -   Если версии несравнимы (конфликт) → запустить стратегию слияния

4.  После успешного слияния: объединить векторы, взяв максимум по
    каждому device_id

**Стратегии слияния по типам данных:**

  ------------------ ----------------- ---------------------------------------------
  Теги (список)      Union + Dedup     Объединение списков, удаление дубликатов. При

                                       удалённо) --- приоритет у более нового

                                       двух устройствах --- сохраняется версия с
                                       более поздним lastModifiedAt

  Настройки          Last-Write-Wins   Побеждает значение с более поздней меткой
  (ключ-значение)                      времени изменения

  Коллекции          Union + Priority  Объединение правил, но приоритет у
  (виртуальные                         пользовательских переименований и ручных
  папки)                               закреплений

  Медиа-метаданные   Field-Level Merge Поле isFavorite --- LWW, поле tags --- Union,
                                       поле noteAttached --- LWW

**Алгоритм слияния для списков тегов:**

1.  Базовое объединение: merged = local_tags ∪ remote_tags

2.  Удаление тегов, помеченных как удалённые на обоих устройствах

3.  Разрешение конфликтов: если тег добавлен локально и удалён удалённо
    --- сравниваются векторные часы, побеждает более новое изменение

4.  При несравнимых версиях --- сохраняются оба варианта, запись
    помечается для ручного разбора

5.  Сортировка результата: пользовательские теги → авто-теги → мета-теги

**8.4 Стратегия резервного копирования**

Система бэкапов обеспечивает защиту данных от потери при сбоях
оборудования, повреждениях БД или ошибочных действиях пользователя.

**Локальные бэкапы (обязательные):**

-   **Частота:** Автоматическое создание каждые 4 часа при активном
    использовании приложения.

-   **Хранение:** Последние 7 версий в зашифрованном формате в
    директории /GlubLink/backups/local/.

-   **Содержимое архива:**

    -   Полная копия базы данных Isar (зашифрована)

    -   Файл настроек пользователя

    -   Локальная база переводов тегов

    -   Журнал изменений (Change Log) за период

    -   Манифест с метаданными: версия схемы, контрольные суммы, список
        устройств

-   **Восстановление:** Пользователь может выбрать любую из сохранённых
    версий для отката. Процесс включает проверку целостности, миграцию
    схемы при необходимости и атомарную замену активной БД.

**Облачные бэкапы (опциональные):**

-   **Активация:** Требует явного согласия пользователя и настройки
    облачного провайдера.

-   **Частота:** Раз в 24 часа или по запросу.

-   **Шифрование:** Данные шифруются на устройстве ключом, производным
    от пароля пользователя (алгоритм PBKDF2). Облачный провайдер не
    имеет доступа к ключам.

-   **Хранение:** 30 дней в облаке с автоматической ротацией.

-   **Восстановление:** Доступно с любого устройства после
    аутентификации. Процесс включает загрузку архива, расшифровку,
    валидацию и применение.

**Экспорт данных (ручной):**

-   **Форматы:** Зашифрованный ZIP-архив или открытый JSON (на выбор
    пользователя).

-   **Содержимое:** Полная копия всех пользовательских данных без
    системных метаданных.

-   **Назначение:** Перенос коллекции на новое устройство, архивация,
    предоставление данных третьим лицам по запросу.

-   **Конфиденциальность:** Пользователь может выбрать, какие данные

-   **Кэширование:** Результаты внешних запросов сохраняются локально с

**8.6 Безопасность данных и шифрование**

Защита пользовательских данных реализуется на нескольких уровнях,
обеспечивая конфиденциальность даже при компрометации одного из
компонентов.

**Шифрование на диске:**

-   **База данных:** Полная база Isar шифруется с использованием
    SQLCipher + AES-256. Ключ шифрования производный от пароля
    пользователя (алгоритм PBKDF2 с 100 000 итераций).

-   **Бэкапы:** Архивы бэкапов шифруются тем же ключом перед записью на
    диск или отправкой в облако.

-   **Управление ключами:** Мастер-ключ никогда не хранится в открытом
    виде. При каждом запуске приложение запрашивает пароль пользователя
    для расшифровки.

**Шифрование сетевого трафика:**

-   **Транспортный уровень:** Все соединения используют TLS 1.3 с
    проверкой сертификатов.

-   **Сквозное шифрование:** Данные синхронизации шифруются на
    устройстве-отправителе и расшифровываются только на
    устройстве-получателе. Облачный сервер выступает только как
    транспорт, не имея доступа к содержимому.

-   **Ключевой обмен:** Используется алгоритм ECDH для безопасного
    обмена сессионными ключами между устройствами.

**Защита секретов:**

-   **API-ключи и токены:** Хранятся в Windows Credential Locker с
    шифрованием через DPAPI. Доступ только после аутентификации
    пользователя в ОС.

-   **Пароли:** Никогда не логируются, не передаются в аналитику, не
    хранятся в конфигурационных файлах.

-   **Маскировка в логах:** Все чувствительные данные (пути к файлам,
    токены, персональные теги) автоматически маскируются перед записью в
    лог.

**Аудит и логирование:**

-   **Что логируется:** События синхронизации, разрешения конфликтов,
    создания бэкапов, изменения прав доступа, ошибки аутентификации.

-   **Что не логируется:** Содержимое заметок и медиафайлов, пути к
    локальным файлам, пользовательские теги без хеширования, пароли и
    токены.

-   **Ротация:** Логи хранятся 90 дней с автоматическим удалением
    устаревших записей. Критические ошибки сохраняются отдельно для
    расследования.

**8.8 Жёсткие правила генерации кода (Синхронизация и данные)**

Данные требования являются обязательными для всех последующих этапов
реализации. Любое отклонение считается архитектурной ошибкой.

1.  **Единый источник состояния для синхронизации.** Все операции с
    данными (добавление, обновление, удаление) выполняются только через
    реактивный провайдер, который управляет очередью синхронизации.
    Запрещено дублировать логику отправки изменений в презентационном
    слое.

2.  **Строгая типизация векторных часов.** Векторные часы описываются
    через специализированный класс VectorClock с методами сравнения и
    слияния. Запрещено использовать Map\<String, int\> напрямую в
    бизнес-логике.

3.  **Атомарность операций с БД.** Любое изменение, затрагивающее
    несколько таблиц или записей, выполняется в транзакции. При ошибке
    происходит полный откат, и пользователь получает понятное сообщение.

4.  **Изоляция побочных эффектов.** Все сетевые запросы, работа с
    облачными API и шифрование инкапсулированы в репозиториях.
    Презентационный слой не импортирует пакеты сетевого взаимодействия
    или криптографии.

5.  **Реактивное обновление интерфейса.** Изменение статуса
    синхронизации (успех, ошибка, прогресс) автоматически уведомляет все
    подписанные виджеты. Запрещено ручное обновление через setState для
    статуса; только через реактивные провайдеры.

6.  **Шифрование на границе данных.** Все данные, покидающие устройство
    (синхронизация, бэкапы, экспорт), шифруются до передачи. Ключи
    шифрования никогда не передаются по сети и не логируются.

7.  **Обработка ошибок на границе фичи.** Каждый модуль системы
    синхронизации имеет собственный обработчик исключений. Ошибки
    логируются, преобразуются в пользовательские сообщения и не приводят
    к падению приложения.

8.  **Доступность интерфейса синхронизации.** Все индикаторы статуса,
    кнопки управления и сообщения об ошибках содержат семантические
    метки, поддерживают навигацию с клавиатуры и имеют достаточную
    контрастность.
